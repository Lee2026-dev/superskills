# 冲突页一键解决设计（去向导 + 批量软链接）

Date: 2026-04-20  
Status: Approved for implementation  
Owner: wenli

## 1. 目标

优化冲突处理体验：

- 移除冲突列表中的“向导处理”入口
- 新增“`一键解决冲突`”批量操作
- 统一按策略选择保留路径，其余路径改为软链接
- 提供可见进度条与完成汇总

## 2. 交互与信息架构

### 2.1 冲突页面

- 在冲突页标题栏新增主按钮：`一键解决冲突`
- 删除每个冲突组 header 中的 `向导处理 →`
- 保留每条路径下的快捷操作（`软链其他` / `删除其他`）作为手动兜底

### 2.2 执行流程

1. 点击 `一键解决冲突`
2. 弹出确认框（说明将批量执行“删除并建立软链接”）
3. 进入执行态：
   - 按钮禁用并显示 `处理中...`
   - 显示顶部进度条
   - 显示 `正在处理 X / Y`
4. 每处理一个 remove->symlink 请求即推进进度
5. 执行结束后展示汇总（成功/失败）
6. 自动刷新扫描结果

## 3. 保留路径选择策略

对每个冲突组（同名 skill）选唯一 `keep_path`：

1. 若存在 `~/.agents/skills/<skill>`，优先保留该路径
2. 若不存在且该组全部来自同一根目录，优先保留 `<root>/skills/<skill>`
3. 若是多根目录混合，按固定优先级链保留：
   - `~/.agents/skills`
   - `~/.codex/skills`
   - `~/.hermes/skills`
   - 其他 `*/skills`
   - 其余路径
4. 若仍并列，按路径字典序最小值兜底

该策略要求：

- 稳定可预测（同输入同结果）
- 可解释（用户可理解选择依据）

## 4. 执行动作

每个冲突组中，除 `keep_path` 外的路径均执行：

- `POST /api/conflict/resolve`
- 参数：`{ keep_path, remove_path, symlink: true }`

注意：

- 默认使用软链接方案，不使用批量删除方案
- 单个请求失败不应中断整体批处理（继续后续项，最终汇总）

## 5. 错误处理与反馈

- 失败项记录：skill 名、remove_path、错误信息
- 完成后 toast 汇总：`成功 N 项，失败 M 项`
- 若 `M > 0`，在页面展示失败明细（最小实现可先 toast + console）
- 下次点击 `一键解决冲突`基于最新扫描结果重试

### 7.1 自动化测试

- 更新/新增 `tests/test_dashboard_app_js.py` 断言：
  - 不出现“向导处理”
  - 出现“一键解决冲突”入口

  - 存在进度状态与更新逻辑

### 7.2 手动验收

- 存在失败时页面不会卡死，并可再次执行重试
